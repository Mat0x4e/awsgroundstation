# AWS Ground Station NOAA-20 demonstrator (v2)

The six issues from v1 have been addressed: the processor role now has `kms:Decrypt`/`kms:GenerateDataKey*`, the output bucket ARN is conditionally passed from the processing pipeline module, the duplicate Ground Station IAM role in the mission profile module is gone (using the security module's role via `groundstation_role_arn` variable), the scheduler re-raises `ClientError` on SNS publish failure, the observability SNS topic now receives a `kms_key_id`, and both Lambdas target `python3.12`. Issue #7 (no Terraform backend) was explicitly accepted for a demo scope.

Watch for: the S3 bucket policy prefix pattern doesn't match the S3 recording config prefix, so Ground Station PutObject calls will be denied by the bucket policy (confirmed); the processor Lambda role lacks `cloudwatch:PutMetricData` permission despite the handler publishing custom metrics on every invocation (confirmed); the scheduler Lambda role has no KMS permissions to publish to its own encrypted SNS topic (likely).

**Verdict**: NEEDS_CHANGES

## High-level view

The S3 recording config writes objects with prefix `year={year}/month={month}/day={day}/satellite={satellite_id}` but the reception bucket policy restricts PutObject to `year=*/month=*/day=*/contact-id=*/*`. The path components don't align — `satellite={satellite_id}` vs `contact-id=*` — so Ground Station's writes will be denied by the bucket policy even though the IAM role grants are correct.

The data processor Lambda calls `cloudwatch_client.put_metric_data()` on every invocation (success and error paths) but its IAM role in the security module grants only S3, SQS, KMS, and CloudWatch Logs permissions. The `PutMetricData` calls will fail with AccessDenied and the errors are swallowed by a bare `except ClientError` in `publish_metrics`, so processing appears to succeed while metrics silently vanish.

The scheduler Lambda publishes to an SNS topic encrypted with the project's KMS CMK. Publishing to a KMS-encrypted SNS topic requires `kms:GenerateDataKey*` and `kms:Decrypt` on the key. The scheduler role has `sns:Publish` but no KMS grants, so the publish call will likely fail with a KMS access denied error — turning the "no suitable pass" notification into the same silent failure pattern that v1 flagged (now the exception is re-raised, which means the Lambda will error out rather than succeed silently, but the operator still doesn't get the notification).

## S3 recording prefix vs bucket policy path

The mission profile's S3 recording config sets the object prefix:

```hcl
prefix = "year={year}/month={month}/day={day}/satellite={satellite_id}"
```

The reception bucket policy constrains PutObject to:

```hcl
Resource = "${aws_s3_bucket.reception.arn}/year=*/month=*/day=*/contact-id=*/*"
```

Ground Station substitutes the template variables at write time, producing keys like `year=2026/month=05/day=22/satellite=43013/...`. The bucket policy expects a `contact-id=` path component at the fourth level. Since the actual key has `satellite=43013` at that position, the policy condition fails and the PutObject is denied. The IAM role's own policy grants unrestricted PutObject on `${bucket_arn}/*`, but S3 bucket policies are evaluated as an additional authorization boundary — both must allow the action.

## Missing cloudwatch:PutMetricData on processor role

The `publish_metrics` function in `lambdas/data_processor/handler.py` is called on both success and error paths:

```python
def publish_metrics(project_name, duration_ms, input_size, status):
    try:
        cloudwatch_client.put_metric_data(
            Namespace=METRICS_NAMESPACE,
            MetricData=[...]
        )
    except ClientError as e:
        logger.error(...)
```

The processor role's IAM policy (`modules/security/main.tf` lines 169-210) grants `s3:GetObject`, `s3:PutObject`, `kms:Decrypt`, `kms:GenerateDataKey*`, `sqs:ReceiveMessage/DeleteMessage/GetQueueAttributes`, and `logs:*`. There is no `cloudwatch:PutMetricData` statement. The `except ClientError` block means the Lambda won't crash, but all custom metrics — processing duration, input size, status counts — will be absent from CloudWatch, making the observability dashboard less useful than it appears.

## Scheduler KMS gap for encrypted SNS publish

The contact notifications topic is encrypted:

```hcl
resource "aws_sns_topic" "contact_notifications" {
  kms_master_key_id = aws_kms_key.groundstation.id
  ...
}
```

The scheduler role grants:

```hcl
{
  Effect = "Allow"
  Action = ["sns:Publish"]
  Resource = aws_sns_topic.contact_notifications.arn
}
```

When a principal publishes to a KMS-encrypted SNS topic, SNS calls KMS on behalf of the publisher. The KMS key policy grants `kms:*` to the account root (which covers all IAM principals in the account via IAM policy evaluation), so the key policy side is satisfied. However, the IAM role itself also needs explicit `kms:GenerateDataKey*` and `kms:Decrypt` grants for the publish to succeed — the KMS key policy's root grant only removes the key-policy gate, it doesn't substitute for the identity policy.

This means the scheduler's "no suitable pass" SNS publish will fail with a KMS AccessDeniedException. Because the v2 fix correctly re-raises the exception, the Lambda will now return an error rather than succeed silently. The fix is to add KMS permissions to the scheduler role.
