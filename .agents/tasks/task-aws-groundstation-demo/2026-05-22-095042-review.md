# AWS Ground Station NOAA-20 demonstrator (v3)

All three issues from v2 have been resolved: the S3 bucket policy prefix now matches the recording config's `satellite={satellite_id}` path component, the processor role grants `cloudwatch:PutMetricData` with resource `*`, and the scheduler role has `kms:GenerateDataKey*` and `kms:Decrypt` on the CMK. The six v1 fixes remain intact.

Watch for: EventBridge cannot publish to the KMS-encrypted `contact_failures` SNS topic because the KMS key policy has no grant for `events.amazonaws.com` — contact failure notifications will be silently dropped (likely).

**Verdict**: APPROVED

## High-level view

One residual gap exists in the observability module: the `contact_failures` SNS topic was encrypted with the project KMS key as part of the v1 fix, but EventBridge publishes to this topic as a service principal. The KMS key policy doesn't grant `events.amazonaws.com` any actions, and service principals don't benefit from the root account's `kms:*` grant the way IAM roles do. EventBridge's publish will fail with a KMS access denied, and EventBridge silently drops undeliverable targets. This is non-blocking for a demo — it only affects the failure notification path and has a straightforward fix (add EventBridge to the KMS key policy) — but the FAILED/FAILED_TO_SCHEDULE alerting won't work until addressed.

## EventBridge publish to KMS-encrypted SNS

When SNS receives a publish request for a KMS-encrypted topic, it calls KMS on behalf of the publisher to generate a data key. The KMS key policy must explicitly allow the publishing service principal to perform `kms:GenerateDataKey*` and `kms:Decrypt`.

The current key policy statements:

```
EnableRootAccountAccess  -> arn:aws:iam::{account}:root  -> kms:*
AllowGroundStationService -> groundstation.amazonaws.com -> Encrypt, Decrypt, GenerateDataKey*, DescribeKey
AllowCloudTrailEncryption -> cloudtrail.amazonaws.com    -> Encrypt, Decrypt, GenerateDataKey*, DescribeKey
```

Service principals don't inherit from the root grant — that mechanism only works for IAM principals whose identity-based policies are evaluated against the key policy's root statement. EventBridge operates as `events.amazonaws.com` with no IAM identity. Without an explicit key policy grant, the KMS call returns AccessDeniedException, the SNS publish fails, and EventBridge drops the event without surfacing the error.

The fix is a fourth key policy statement:

```hcl
{
  Sid    = "AllowEventBridgeEncryption"
  Effect = "Allow"
  Principal = {
    Service = "events.amazonaws.com"
  }
  Action = [
    "kms:GenerateDataKey*",
    "kms:Decrypt"
  ]
  Resource = "*"
}
```

This is non-blocking for demo purposes: the failure path (FAILED/FAILED_TO_SCHEDULE contacts) is an operational nicety, not a data-flow dependency. The core reception and processing paths are unaffected.
