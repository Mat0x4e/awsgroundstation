# AWS Ground Station NOAA-20 demonstrator (Terraform + Lambda)

A greenfield Terraform project that stands up an AWS Ground Station S3 Data Delivery pipeline for NOAA-20 X-band HRD reception: KMS encryption, IAM roles, an S3 reception bucket with Glacier Deep Archive lifecycle, a mission profile targeting 7812 MHz / 30 MHz BW / QPSK / RHCP, an EventBridge-triggered Lambda that schedules the highest-elevation weekday pass, and a conditional SQS-to-Lambda processing pipeline for CADU frame validation. Observability is handled by a CloudWatch dashboard, failure alarms, and EventBridge rules routing contact state changes to SNS.

Watch for: the processor Lambda IAM role has no KMS decrypt permission so reads from the KMS-encrypted reception bucket will fail (confirmed); two competing Ground Station IAM roles with overlapping S3 write policies create confusion about which role the service actually uses (confirmed); the scheduler Lambda swallows SNS publish failures silently, meaning "no pass" alerts can vanish without any signal (confirmed); the contact-failures SNS topic in the observability module is unencrypted at rest unlike every other SNS topic in the project (confirmed).

## High-level view

The security module creates a single KMS CMK, three IAM roles, a CloudTrail trail, and an SNS notifications topic. The KMS key policy grants scoped encrypt/decrypt to the Ground Station and CloudTrail services, but the processor Lambda role never receives KMS decrypt permission — a gap that will cause S3 GetObject calls to fail at runtime.

The mission profile module introduces a second Ground Station IAM role (`groundstation_delivery`) with the same S3 PutObject/GetBucketLocation and KMS Encrypt permissions as the `groundstation` role in the security module. Only the delivery role is wired into the S3 recording config, making the security module's Ground Station role unused dead weight.

The scheduler Lambda paginates available contacts, filters to weekdays above minimum elevation, and reserves the best pass. Its operational weakness is that SNS publish failures in the "no pass" notification path are caught and logged but don't propagate — the Lambda returns success even though an operator alert was lost.

The processing pipeline is conditionally deployed behind `var.enable_processing_pipeline`. The `output_bucket_arn` passed to the security module is hardcoded to `""` in `main.tf`, so the processor role's PutObject permission falls back to the reception bucket — but writes actually land in a different bucket (`output`) that the role has no grant for.

The observability module's `contact_failures` SNS topic is the only unencrypted topic in the project, an inconsistency that may violate compliance expectations or SCPs enforcing encryption at rest.

1. **Processor role missing KMS Decrypt** — The processor Lambda role in `modules/security/main.tf` has no `kms:Decrypt` permission. Add a statement granting `kms:Decrypt` and `kms:GenerateDataKey*` on the KMS key ARN so `s3:GetObject` succeeds on the SSE-KMS reception bucket.

3. **Duplicate Ground Station IAM role** — Both `modules/security/main.tf` and `modules/mission_profile/main.tf` create a Ground Station service role with identical S3 permissions. Only the mission profile's `groundstation_delivery` role is referenced by the S3 recording config. Remove or consolidate the unused role in the security module.
4. **Silent SNS publish failure in scheduler** — `publish_no_pass_notification` catches `ClientError` and logs it but doesn't re-raise. If the publish fails, the Lambda returns success with `scheduled: False` and the operator never learns. Re-raise or return a distinct error code.
5. **Unencrypted contact-failures SNS topic** — `modules/observability/main.tf` creates `aws_sns_topic.contact_failures` without `kms_master_key_id`, unlike the topics in the security and processing modules. Add KMS encryption for consistency and compliance.
6. **Python 3.9 runtime end-of-life** — Both Lambdas target `python3.9`, which reached standard support EOL in AWS Lambda. Use `python3.12` or later.
7. **No Terraform backend configuration** — State is local by default. For a multi-operator or CI/CD workflow, configure an S3 + DynamoDB backend or Terraform Cloud. Acceptable for a demo, but worth noting for production path.

## Processor role permission gap

The processor Lambda needs `s3:GetObject` on the KMS-encrypted reception bucket. The IAM policy in `modules/security/main.tf` grants `s3:GetObject` but includes no `kms:Decrypt` statement. Because the reception bucket enforces SSE-KMS with bucket keys, every GetObject call requires decrypt permission on the CMK. The Lambda will fail with `AccessDenied` from KMS.

The PutObject resource for the processor role is conditional:

```hcl
Resource = var.output_bucket_arn != "" ? "${var.output_bucket_arn}/*" : "${var.reception_bucket_arn}/*"
```

In `main.tf`, the security module always receives `output_bucket_arn = ""` even when the processing pipeline is enabled. The actual output bucket is created inside the processing pipeline module, so the processor role's PutObject grant targets the reception bucket rather than the output bucket. Writes to the output bucket will be denied.

## Duplicate Ground Station delivery role

The security module creates `aws_iam_role.groundstation` with S3 PutObject + KMS Encrypt. The mission profile module creates `aws_iam_role.groundstation_delivery` with identical permissions. Only `groundstation_delivery` is wired into `awscc_groundstation_config.s3_recording`. The security module's role is orphaned — never referenced, never assumed.

## Scheduler silent failure path

```python
def publish_no_pass_notification(sns_topic_arn, message):
    try:
        sns_client.publish(...)
    except ClientError as e:
        logger.error(...)
```

The caller doesn't check the return value, so the handler returns `{"scheduled": False, "reason": "no_suitable_pass"}` with status 200. An operator expecting to receive an SNS alert when no pass is scheduled has no signal if the publish itself fails (KMS issues on the encrypted topic, permission changes). Logs are passive — nobody is alarmed by a missing alarm.

## Observability SNS encryption inconsistency

```hcl
resource "aws_sns_topic" "contact_failures" {
  name = "${var.project_name}-${var.environment}-contact-failures"
  # no kms_master_key_id
}
```

The `contact_notifications` topic (security module) and `processing_failures` topic (processing pipeline) both use KMS encryption. This topic carries satellite ARNs, contact IDs, and timing data unencrypted at rest. SCPs or AWS Config rules enforcing encryption at rest will block creation or flag non-compliance.

## CADU validation: zero-byte file handling

The data processor treats any file shorter than 1024 bytes as `validation_status: "too_short"` and still writes a JSON metadata record to the output bucket as a successful result. A zero-byte file from a cancelled contact passes through without error and produces output indistinguishable (at the status level) from a legitimate short capture. For production, the processor should reject or quarantine zero-byte/empty objects rather than treating them as valid processing targets.

## SQS visibility timeout margin

The processing queue sets `visibility_timeout_seconds = 360` against a Lambda timeout of 300 seconds. The 60-second margin means that if the Lambda approaches its timeout, the message could become visible again before Lambda reports success, causing duplicate processing. AWS recommends at least 6x the function timeout. The current workload is lightweight so the practical risk is low, but the margin will become insufficient if processing grows heavier.
